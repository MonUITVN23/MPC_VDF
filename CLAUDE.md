 SYSTEM PROMPT & PROJECT CONTEXT: Hybrid VDF-MPC Protocol for Multi-Bridge Cross-chain DAOs

## 1. Project Overview & Academic Goals
You are assisting in the development of a Proof-of-Concept (PoC) for a highly secure, decentralized, and cross-chain Verifiable Random Function (VRF) protocol [1]. This system solves the "Randomness Paradox" in distributed networks, specifically targeting multi-chain Decentralized Autonomous Organizations (DAOs) [2, 3].

The system implements a **Defense-in-Depth (Dual-layer)** security model:
1.  **Multi-Party Computation (MPC):** Prevents **Input Bias**. A collective seed is generated by a committee threshold, ensuring no single entity can manipulate the initial randomness [4].
2.  **Verifiable Delay Function (VDF):** Prevents **Premature Prediction (Front-running)**. Forces a sequential time delay $T$ before the final result is known, making the MPC committee's early knowledge useless [5, 6].

### Core Philosophies (CRITICAL)
-   **Bridge-Agnostic Verification:** Bridges are treated as transport layers only. The destination application must never trust payloads blindly and must validate cryptographic conditions before finalization [7].
-   **Optimistic Verification:** To avoid high gas costs for mandatory on-chain VDF checks, destination contracts use an optimistic queue. Expensive verification runs only when challenged during a challenge window [8, 9].
-   **Keep Research Direction Stable:** Preserve the MPC + VDF thesis and benchmarking methodology; update only implementation details that are outdated versus the current repo architecture.

## 4. End-to-End Workflow & Architecture (Updated)
Implement this sequence and avoid redundant cross-chain messages:

1.  **Request (Source Chain):** User calls `requestRandomness(seed_user)` on `RandomRouter.sol`. Contract emits `LogRequest`.
2.  **MPC Generation (Off-chain Rust):** Committee derives `seed_collective` and aggregate signature $\sigma$ via threshold flow.
3.  **VDF Evaluation (Off-chain Executor):** Compute $y = x^{2^T} \pmod \Delta$ and proof $\pi$ from `seed_collective`.
4.  **Relay via Router + Adapter:** Executor calls `relayVDFPayload(...)` on router with `bridgeId: bytes32`; router dispatches payload through selected adapter.
5.  **Destination Optimistic Handling:** Receiver decodes payload, performs required admission checks (including optional BLS mode), enqueues result, and opens challenge window.
6.  **Challenge/Finalize:** Anyone can challenge during the window; if unchallenged and valid by policy, finalize randomness for consumer dApps.

> Legacy note: old `RandomSender + uint8 bridge_id + SMC-centric relay` assumptions are deprecated in current architecture and must not be reintroduced.

## 5. Mathematical Foundations (Rust Crypto Engine)
When writing off-chain Rust logic, apply:
-   **VDF Evaluation:** Strictly sequential squaring; no parallel shortcuts [15].
    -   $y = x^{2^T} \pmod \Delta$ where $\Delta$ is a large negative prime discriminant.
-   **Proof Generation (Wesolowski):** Let $2^T = q \cdot \ell + r$ [16].
    -   $\pi = x^q \pmod \Delta$.
    -   Derive prime $\ell$ using Fiat-Shamir heuristic.

## 6. Strict Coding Guidelines (Rules for AI Generation)

### A. Solidity (Smart Contracts)
-   **Gas-sensitive VDF Verify:** For heavy modular exponentiation, prefer optimized/precompile-based paths (`0x05`) over naive big-int loops [17].
-   **Optimistic State Machine:** Receiver logic must preserve clear submit/challenge/finalize states and explicit deadlines.
-   **Router/Adapter Compliance:** New source-chain relay code must target `RandomRouter` + `IBridgeAdapter` abstraction; do not hardwire protocol-specific business logic into router core.

### B. Cross-chain Bridge Implementation
-   **Axelar/LayerZero/Wormhole:** Implemented as adapters behind common `IBridgeAdapter`.
-   **New bridges:** Add as new adapter contract + deploy config entry; avoid modifying core request workflow.
-   **Backward compatibility:** Legacy placeholder interfaces may remain for compatibility/research notes, but production path follows router/adapters.

### C. Rust (Off-chain Nodes)
-   **Modular Design:** Keep strict separation:
    -   `network_module`: event listening, bridge routing, relay orchestration.
    -   `crypto_engine`: MPC/DKG + VDF + proof logic.
-   **VDF Implementation:** `vdf-rs` with IQCG only (RSA-based VDF prohibited).
-   **Configurable Threshold:**
    ```rust
    const DEFAULT_TOTAL_NODES: usize = 4;
    const DEFAULT_THRESHOLD: usize = 3;
    ```
-   **Failover-aware Relay:** Multi-bridge routing should support priority/fallback behavior and bridge-tagged telemetry.
