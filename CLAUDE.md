 SYSTEM PROMPT & PROJECT CONTEXT: Hybrid VDF-MPC Protocol for Cross-chain DAOs

## 1. Project Overview & Academic Goals
You are assisting in the development of a Proof-of-Concept (PoC) for a highly secure, decentralized, and cross-chain Verifiable Random Function (VRF) protocol [1]. This system solves the "Randomness Paradox" in distributed networks, specifically targeting multi-chain Decentralized Autonomous Organizations (DAOs) [2, 3]. 

The system implements a **Defense-in-Depth (Dual-layer)** security model:
1.  **Multi-Party Computation (MPC):** Prevents **Input Bias**. A collective seed is generated by a committee threshold, ensuring no single entity can manipulate the initial randomness [4].
2.  **Verifiable Delay Function (VDF):** Prevents **Premature Prediction (Front-running)**. Forces a sequential time delay $T$ before the final result is known, making the MPC committee's early knowledge useless [5, 6].

### Core Philosophies (CRITICAL)
-   **Bridge-Agnostic Verification:** Cross-chain bridges (like Axelar) are treated purely as "dumb couriers" [7]. The Destination Smart Contract must NEVER blindly trust the bridge's payload. It must independently verify the mathematical proof [7].
-   **Optimistic Verification:** To bypass exorbitant gas fees for On-chain VDF verification, the destination chain defaults to an "Optimistic Rollup" style queue. Mathematical verification is ONLY executed on-chain if a dispute is raised during a Challenge Window [8, 9].

## 3. End-to-End Workflow & Architecture
Implement the following exact sequence. **Note: Do NOT implement redundant cross-chain messages.** The Source Chain contract does NOT send the commitment directly to the Destination Chain via the bridge; everything is bundled by the Off-chain Executor to save gas [7, 12, 13].

1.  **Request (Source Chain - Sepolia):** User calls `requestRandomness(seed_user)` on `RandomSender.sol`. The contract emits a `LogRequest` event [12].
2.  **MPC Generation (Off-chain Rust Nodes):** The dynamic MPC committee listens to the event. They run a Pedersen Distributed Key Generation (DKG) over BLS12-381 to generate a `seed_collective` and aggregate a collective signature $\sigma$ [12, 14].
3.  **VDF Evaluation (Off-chain Rust Executor):** One node acts as the Executor. It takes the `seed_collective` ($x$) and computes the Wesolowski VDF over $T$ iterations: $y = x^{2^T} \pmod \Delta$. It also generates a concise proof $\pi$ [12, 13, 15].
4.  **Cross-chain Relay (Axelar GMP):** The Executor bundles everything into a SINGLE "All-in-One Package": `{requestID, y, \pi, seed_collective, \sigma}` and sends it via Axelar GMP [13].
5.  **Optimistic Handling (Destination Chain - Amoy):** `RandomReceiver.sol` receives the payload. It first verifies $\sigma$ using the MPC Public Key. If valid, it pushes $\{y, \pi\}$ into a Challenge Queue rather than verifying the VDF immediately [9, 13].

## 4. Mathematical Foundations (For Rust Crypto Engine)
When writing the Rust off-chain engine, apply these formulas:
-   **VDF Evaluation:** Must be strictly sequential. Do not attempt to parallelize the squaring loop [15].
    -   Calculation: $y = x^{2^T} \pmod \Delta$ (where $\Delta$ is a large negative prime discriminant) [15].
-   **Proof Generation:** Compute quotient $q$ and remainder $r$ such that $2^T = q \cdot \ell + r$ [16].
    -   Proof $\pi = x^q \pmod \Delta$ [16].
    -   Use the **Fiat-Shamir heuristic** to derive the prime $\ell$ from the input [16].

## 5. Strict Coding Guidelines (Rules for AI Generation)

### A. Solidity (Smart Contracts)
-   **Extreme Gas Optimization for ModExp:** When writing the `verifyVDF()` function, **DO NOT** use standard Solidity BigInt math libraries. You **MUST** write inline Assembly (Yul) to use the **Ethereum Precompiled Contract `0x05`** (`ModExp`). This reduces verification gas from millions to just a few thousand units [17].
    -   *Reference Signature:* `(bool success, bytes memory result) = address(0x05).staticcall(...)` [17].
-   **Optimistic Rollup Logic:** `RandomReceiver.sol` must implement a queue data structure. Generate the following state-machine functions [9]:
    1.  `submitOptimisticResult()`: Accepts the Axelar payload, checks the MPC signature, and starts the `CHALLENGE_WINDOW` (10 minutes) timer.
    2.  `challengeResult()`: Allows any external caller to flag a fraudulent result. Requires `STAKE_AMOUNT` (1 ether) deposit. This is the ONLY function that calls the expensive `verifyVDF()` (the `0x05` precompile) [9].
    3.  `finalizeRandomness()`: Executed if `CHALLENGE_WINDOW` passes with no disputes. Delivers the final random number to the consuming dApp.
    4.  `slashAndReward()`: If challenge succeeds (VDF invalid), slash the malicious submitter and reward the challenger.
